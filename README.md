# Cell Counts Dashboard

## Running the Project

All commands should be run from the repository root. The project is designed to run in GitHub Codespaces via the 'Makefile' with Python 3 available.

```bash
make setup      # install dependencies
make pipeline   # initialize & populate db, generate all outputs
make dashboard  # Start the Streamlit dashboard @ localhost:8501
```

The pipeline must be run before the dashboard, as the dashboard reads from pre-computed output files.

## Code Structure

The repository is structured in alignment with the three main functions it needs to fulfil: schema setup, ETL processing, and dashboard display.

**`load_data.py`** handles database initialization and data loading. 
- Drops and recreates the database on each run to guarantee a clean state
- Initializes table schemas
- Loads data from CSV, transforms, and inserts into tables.
- Validates row counts after loading

**`queries.py`** Contains SQL as named string constants to separate SQL and Python logic. Promotes consistency if queries are used in multiple settings.

**`pipeline.py`** Performs all querying, computation, and transformation necessary for dashboard displays.
- Queries db for population data
- Performs aggregations and transformations on selected populations.
- Performs statistical tests and generates plots.
- Exports results as CSV files to `exports/`, where they can be read by the dashboard.

**`dashboard.py`** creates the dashboard in the users' browser and displays the results generated by the pipeline. Each section is a self-contained display function that reads one or more CSVs and renders a Streamlit widget. `@st.cache_data` is used on all data loading functions to avoid redundant disk reads on widget interaction.

## Database Schema

Three tables model the data:

**`subjects`** - one row per patient
| Column | Type | Notes |
|---|---|---|
| subject_id | TEXT | Primary key |
| condition | TEXT | e.g. melanoma, carcinoma |
| age | INT | |
| sex | TEXT | |
| treatment | TEXT | e.g. miraclib |
| response | TEXT | yes / no |
| project | TEXT | |

**`samples`** - one row per biological sample
| Column | Type | Notes |
|---|---|---|
| sample_id | TEXT | Primary key |
| subject_id | TEXT | Foreign key references subjects |
| sample_type | TEXT | e.g. PBMC |
| time_from_treatment_start | INT | Days from treatment start |

**`cellCounts`** - one row per (sample, cell population) pair
| Column | Type | Notes |
|---|---|---|
| sample_id | TEXT | Composite PK with population |
| population | TEXT | e.g. b_cell, cd8_t_cell |
| count | INT | Raw cell count |

### Design Rationale

Subject-level attributes (age, sex, condition, treatment, response, project) are stored once in **subjects** rather than repeated on every sample row. This eliminates redundancy by establishing a one-to-many relationship between subects and samples.
If more complex subject-sample relationships were to exist (i.e. where age varies in a longitudinal study), then some subject fields would need to be moved to the sample level.

Cell counts use a long/tidy format in `cellCounts` rather than five separate count columns on `samples`. Adding a new cell population requires no schema change, as it is simply a new value in the `population` column.Queries that aggregate across populations (e.g. computing total count per sample) work uniformly regardless of how many populations exist.

An additional reason to separate out the cell counts to a new table is it enable's much more flexibility in what can be included with a sample. Suppose some entirely new measure is added to the study (such as the subject's temperature & weight at the time of the sample). Following the patern taken with **cellCounts**, a new table could be added to store this data without having to alter the **subjects** or **samples** schema. This allows for some limited backwards compatability as well as separation of data categories.

### Scalability

| Dimension | Implication |
|---|---|
| More projects | `project` on `subjects`; filter or GROUP BY as needed |
| More samples | `samples` scales linearly. |
| More subjects | `subjects` scales linearly. |
| More cell populations | New rows in `cellCounts`, no schema change required |
| More analytics | Join `subjects`, `samples`, and `cellCounts` to get any combination of subject metadata, sample metadata, and cell counts in one query |
| Very large data | Should add indexes on `cellCounts(sample_id)` and `samples(subject_id)` to improve join performance. |



## Dashboard

The dashboard displays four sections in sequence:

1. **Cell Population Frequencies** - filterable table of relative cell frequencies per sample (Part 2)
2. **Boxplots** - per-population frequency distributions comparing responders vs non-responders (Part 3)
3. **Statistical Analysis** 
- Student's t-test results with Bonferroni correction across 5 populations (Part 3)
- Q-Q plots confirming the normality assumption for the t-test (Part 3)
4. **Baseline Subset Analysis** - sample and subject counts for the melanoma/miraclib/PBMC/baseline cohort broken down by project, response, and sex (Part 4)

### Interpretation of Statistics
A main takeaway from the results is that CD4 T cell frequency shows a statistically significant difference between responders and non-responders (corrected p = 0.025), suggesting it may be a predictor of miraclib response. However, without performing the multi-test correction, this result is no longer significant.
Although QQ plots show that normality is reasonable, it could also be interesting to perform a non-parametric test. If the sample size were smaller, then this would likely be more appropriate. In this case, the Wilcoxin Rank-Sum test would be the best choice. The two-sided version tests whether one group's variable is systematicaly higher or lower than the other. This directly ties into the question about whether any differences in cell population frequencies could be a predictor of response.
